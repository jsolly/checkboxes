# Decision Points And Vibe Complexity Design

## Goal

Show two complexity measurements on each framework card:

- **Decision Points**: a deterministic, cross-paradigm metric derived from the actual checkbox implementation source.
- **Vibe Complexity**: the existing AI-judged complexity score, renamed and refactored to analyze the same implementation source.

The feature should make the public comparison more honest and auditable. Decision Points replaces the original request for a deterministic cyclomatic complexity number because strict McCabe cyclomatic complexity is not comparable across JavaScript, framework templates, declarative attribute DSLs, and CSS-only implementations.

## Background

The current stats pipeline stores one `complexityScore` in `src/data/framework-stats.json`. That score is generated by Gemini in `src/utils/evaluateComplexity.ts` and displayed by `src/components/shared/PerformanceBadge.astro`.

The stats generator currently finds files by looking for a filename containing `container` inside each framework directory. That means the AI scorer can evaluate display wrapper files like `ReactContainer.astro` instead of the actual nested checkbox implementation displayed in the card.

The new design fixes both problems:

- Replace the ambiguous `complexityScore` concept with explicit deterministic and AI-judged metrics.
- Use the actual implementation files for both metrics, not `*Container.astro` wrappers.

## Definitions

### Decision Points

Decision Points is an integer count of branching, looping, or conditional constructs in a framework implementation. It starts at `0`, not `1`, because it counts decision constructs rather than McCabe paths.

Decision Points is not cyclomatic complexity. It is McCabe-inspired, but defined for this cross-framework comparison so every implementation can receive an auditable deterministic number.

Examples of constructs that count:

- JavaScript control flow: `if`, `else if`, ternaries, non-default `switch` cases, loops, `catch`, and short-circuit operators.
- Framework templates: Vue `v-if` / `v-else-if` / `v-show` / `v-for`, Svelte `{#if}` / `{:else if}` / `{#each}` / `{#await}` / `{#key}`.
- Declarative attributes: Alpine `x-if` / `x-show` / `x-for`, Datastar condition/loop attributes, and Hyperscript conditional or loop tokens.
- CSS behavior selectors: `:has()`, `:is()`, `:not()`, `:where()`, `:checked` sibling behavior, and behavioral state selectors.

Examples of constructs that do not count:

- Function declarations, class declarations, imports, comments, whitespace, or formatting.
- JSX nesting depth.
- Pure presentation CSS.
- JavaScript higher-order method calls such as `.map()` or `.every()` unless their inline callback contains counted control flow.

### Vibe Complexity

Vibe Complexity is the AI-judged 0-100 score that exists today. It should remain separate from Decision Points and must be labeled as model-judged. The score captures qualities deterministic counts miss, such as idiomatic style, naming, indirection, and surprise factor.

## Requirements

1. Each framework card displays Decision Points and Vibe Complexity.
2. Decision Points are deterministic and generated without an AI model.
3. Vibe Complexity uses the existing Gemini-based scoring path, renamed to make the AI dependency explicit.
4. Both metrics analyze the actual checkbox implementation source shown in the card's code block.
5. Bundle size behavior remains unchanged.
6. Sorting supports bundle size, Decision Points, and Vibe Complexity separately.
7. Documentation explains why the deterministic metric is not labeled cyclomatic complexity.
8. Generation fails loudly if deterministic analysis cannot parse or analyze an implementation file.
9. If `GEMINI_API_KEY` is missing or Gemini rate-limits, generation preserves existing Vibe Complexity values when available.

## Architecture

Add a small stats pipeline layer with three responsibilities:

1. **Implementation source resolution**: choose the source file for each framework.
2. **Decision Point analysis**: parse source and count deterministic decision constructs.
3. **Vibe Complexity evaluation**: send the same source to Gemini and preserve old values when AI scoring is unavailable.

### Implementation Source Resolver

The generator should stop discovering implementation files by looking for `container` in filenames. Instead, each framework should resolve to the actual implementation source displayed in its card.

The preferred design is an explicit source path in framework configuration, for example:

```ts
export const FRAMEWORKS = {
  react: {
    displayName: "React",
    implementationFile: "src/components/react/ReactNestedCheckboxes.jsx",
  },
  cssOnly: {
    displayName: "CSS Only",
    implementationFile: "src/components/cssOnly/cssOnly.astro",
  },
} as const;
```

The resolver should read that file and return the framework id, absolute path, relative path, extension, and source text. Containers can continue importing `?raw` for display, but the stats generator should use the shared config as the source of truth.

### Decision Point Analyzer

Expose one public analyzer entry point, such as:

```ts
analyzeDecisionPoints(source: ImplementationSource): DecisionPointResult
```

The analyzer dispatches by extension:

- `.js`, `.jsx`, `.ts`, `.tsx`: parse with a JavaScript/TypeScript AST parser and count language control-flow constructs.
- `.vue`: parse the SFC, analyze script blocks as JavaScript, and count Vue template directives.
- `.svelte`: parse component scripts as JavaScript and count Svelte template blocks.
- `.astro`: analyze frontmatter and script content as JavaScript, then count relevant template directives and attributes.
- HTML-like source: extract inline scripts and count declarative condition/loop attributes.
- CSS-only behavior inside Astro or CSS blocks: count behavioral selector constructs only.
- Hyperscript bodies: use a deterministic token counter for conditional and loop keywords because there is no stable public AST API.

The result should include both total and breakdown so the number is auditable:

```ts
interface DecisionPointResult {
  value: number;
  normalizedScore: number;
  breakdown: {
    jsControlFlow: number;
    templateDirectives: number;
    selectors: number;
    declarativeAttrs: number;
  };
}
```

Normalize with a fixed cap:

```ts
normalizedScore = Math.min(100, Math.round((decisionPoints / 30) * 100));
```

The raw count is the durable value. The normalized score is only for at-a-glance display.

### Vibe Complexity Evaluator

Rename the current AI evaluator from complexity language to Vibe Complexity language. Its prompt should explicitly state that it is scoring implementation complexity on a 0-100 model-judged scale and should receive the same implementation sources used by the Decision Point analyzer.

The existing median-over-multiple-runs behavior can remain. If no API key is available, or if Gemini rate-limits after collecting no new result for a framework, the generator should preserve the previous Vibe Complexity value from `framework-stats.json`.

## Data Shape

Replace the current ambiguous `complexityScore` / `complexityZScore` fields with explicit metric fields.

Recommended `FrameworkStats` shape:

```ts
interface FrameworkStats {
  bundleSize: number;
  decisionPoints: number;
  decisionPointScore: number;
  vibeComplexity: number;
  bundleSizeZScore: number;
  decisionPointZScore: number;
  vibeComplexityZScore: number;
  decisionPointBreakdown: {
    jsControlFlow: number;
    templateDirectives: number;
    selectors: number;
    declarativeAttrs: number;
  };
}
```

The JSON metadata should describe each metric, including the Decision Points cap. The generator does not need to preserve backwards compatibility with the old `complexityScore` shape because this is an in-progress project and the public code should describe the current metric model directly.

## Card UI

Each card should display all three comparison metrics:

- `Decisions: 11 (37/100)`
- `Vibe: 42/100`
- `JS Bundle: 1.53kb`

Decision Points and Vibe Complexity should have separate tooltips. The Decision Points tooltip should emphasize deterministic analysis and link to the methodology document or page. The Vibe tooltip should mention that the score is AI-judged and may change when the model changes.

The current badge color system can continue using z-scores, with higher z-scores treated as worse for all three metrics.

## Sorting

The sort control should expose separate options:

- Bundle Size, smallest and largest first.
- Decision Points, fewest and most first.
- Vibe Complexity, simplest and most complex first.

The old generic Complexity sort should be removed instead of kept as a compatibility alias.

## Documentation

Add public methodology documentation explaining:

- What is being compared.
- Why strict cyclomatic complexity is not displayed.
- The Decision Points rules.
- How Vibe Complexity is generated.
- How scores are normalized.
- Known limitations.
- How to reproduce the numbers.

The methodology can start as `METHODOLOGY.md` and be linked from the card tooltip or page copy. A future implementation may turn it into a first-class Astro page if that fits the site better.

## Error Handling

Deterministic analysis should fail the stats generation command when:

- The configured implementation file is missing.
- The file extension is unsupported.
- Parsing fails.
- The analyzer cannot produce a Decision Point result.

Vibe Complexity should be tolerant of external model availability:

- If `GEMINI_API_KEY` is missing, preserve existing Vibe Complexity values.
- If Gemini rate-limits after some runs, use collected measurements where available and preserve prior values for frameworks without new measurements.
- Non-rate-limit Gemini errors should still fail generation.

## Testing

Add focused tests for:

- JavaScript control-flow counting, including ternaries and short-circuit operators.
- JSX conditional rendering.
- Vue, Svelte, and Astro template/directive counting.
- CSS behavioral selector counting.
- Hyperscript or declarative attribute token counting.
- Source resolution proving each framework points at the displayed implementation, not the container.
- Stats generation JSON shape and z-score fields.
- Preservation of existing Vibe Complexity when no Gemini key is present.

Run lint and build as the final verification for implementation.

## Acceptance Criteria

- Each framework card shows Decision Points and Vibe Complexity as separate metrics.
- Existing AI complexity wording is renamed to Vibe Complexity in code, generated data, UI, tooltips, and docs.
- Decision Points are generated deterministically from implementation files.
- Both metrics use the actual checkbox source displayed in the card.
- Sorting distinguishes Decision Points from Vibe Complexity.
- Public methodology documentation exists and explains that Decision Points is not cyclomatic complexity.
- `pnpm build` succeeds after the implementation.
